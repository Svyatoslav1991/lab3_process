# lab3_process

Учебный проект на **Qt/C++**, демонстрирующий запуск процессов, взаимодействие между **parent/child** приложениями через **QProcess**, а также обмен данными через **QSharedMemory**. Проект собран на **CMake**, реализован на **Qt Widgets** и покрывает ключевую логику **unit-тестами на Qt Test**.

## Что показывает проект

Проект фокусируется на трёх практических сценариях межпроцессного взаимодействия:

- запуск процесса в detached-режиме через `QProcess::startDetached()`;
- синхронный запуск через `QProcess::execute()`;
- управляемый запуск дочернего процесса через `QProcess::start()` с последующим обменом данными;
- передача строковых сообщений между процессами через `stdin/stdout`;
- обмен строковыми данными через сегмент разделяемой памяти `QSharedMemory`;
- изоляция общей логики работы с shared memory в отдельный переиспользуемый модуль.

Проект удобно использовать как учебный пример по темам:

- `QProcess`;
- `QSharedMemory`;
- IPC в Qt;
- организация Qt/CMake-проекта;
- базовое покрытие GUI и service-логики unit-тестами.

## Архитектура

Репозиторий разделён на три независимых части:

### `parent`
Родительское GUI-приложение, которое:

- запускает дочерний процесс разными способами;
- отправляет данные в дочерний процесс;
- принимает данные из `stdout` дочернего процесса;
- управляет созданием, записью, чтением и отключением shared memory.

### `child`
Дочернее GUI-приложение, которое:

- отправляет данные родителю через `stdout`;
- принимает данные от родителя через `stdin`;
- подключается к уже созданному сегменту shared memory;
- записывает и считывает строковые данные из общей памяти.

### `common`
Общий модуль с классом `SharedMemoryChannel`, который инкапсулирует:

- создание и подключение к сегменту `QSharedMemory`;
- безопасную блокировку памяти через RAII;
- сериализацию и десериализацию `QString` через `QDataStream`;
- проверку внутреннего формата данных по заголовку, сигнатуре и версии протокола.

## Ключевые технические решения

### 1. Разделение по ответственности
Общая логика вынесена в `common`, а UI и сценарии взаимодействия разделены между `parent` и `child`. Это делает код проще для сопровождения и тестирования.

### 2. Shared memory через собственный протокол
Для записи данных в `QSharedMemory` используется небольшой служебный заголовок:

- `magic` — сигнатура формата;
- `version` — версия протокола;
- `payloadSize` — размер полезной нагрузки.

Такой подход позволяет:

- валидировать содержимое памяти;
- защититься от чтения повреждённых данных;
- расширять формат без слома совместимости.

### 3. Сериализация через `QDataStream`
Строка не копируется в память «как есть», а сериализуется в `QByteArray` через `QDataStream`. Это даёт стабильный и контролируемый двоичный формат.

### 4. Безопасная блокировка `QSharedMemory`
Для доступа к общей памяти используется RAII-обёртка `LockGuard`, которая автоматически вызывает `lock()/unlock()` и делает код чтения/записи надёжнее.

### 5. Построчная обработка stdout
В `parent` реализована буферизация `stdout` дочернего процесса с корректной обработкой частичных фрагментов строки. Это важно для реального IPC, когда данные приходят не всегда целиком одним куском.

## Unit-тесты

В проекте предусмотрены unit-тесты на **Qt Test** для логики, которую имеет смысл проверять автоматически.

Покрываются, в частности:

- `SharedMemoryChannel`:
  - создание, подключение и отключение shared memory;
  - запись и чтение строк;
  - работа с Unicode;
  - пограничные случаи и ошибки протокола;
- `child/MainWindow`:
  - инициализация UI;
  - работа с shared memory через интерфейс;
  - отправка данных в `stdout`.

Тесты написаны так, чтобы проверять не только happy path, но и защиту от некорректного состояния данных.

## Сборка

### Требования

- C++17
- CMake 3.16+
- Qt 5 или Qt 6
- модули Qt:
  - `Core`
  - `Widgets`
  - `Test`

### Сборка через CMake

```bash
git clone <your-repository-url>
cd lab3_process
cmake -S . -B build
cmake --build build
```

### Запуск тестов

```bash
ctest --test-dir build --output-on-failure
```

## Что полезно посмотреть в коде

Если открывать проект как учебный пример, я бы рекомендовал смотреть в таком порядке:

1. `common/app/include/sharedmemorychannel.h`
2. `common/app/src/sharedmemorychannel.cpp`
3. `child/app/src/mainwindow.cpp`
4. `parent/app/src/mainwindow.cpp`
5. тесты в `common/tests` и `child/tests`

Так проще понять, как устроен низкоуровневый IPC-слой и как он используется в GUI.
